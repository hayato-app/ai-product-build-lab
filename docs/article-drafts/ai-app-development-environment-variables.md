## 結論

環境変数は、APIキーや接続先URLのような「コードに直接書きたくない設定値」を、アプリの外側から渡すための仕組みです。

AIアプリ開発では、OpenAI APIキーなどの秘密情報を扱う場面が多いため、環境変数の考え方を早い段階で理解しておくと安全です。

初心者のうちは、まず次の3つを押さえれば十分です。

| 覚えること | 意味 |
| --- | --- |
| APIキーをコードに直接書かない | GitHubやブラウザに漏れるリスクを避ける |
| ローカルでは `.env` を使う | 手元の開発環境だけで使う設定値を置く |
| 本番環境ではVPSやホスティング側で設定する | 公開サーバーで安全に設定値を渡す |

![AIアプリ開発で、.envファイル、鍵、サーバー、AI APIが安全につながるイメージ図](/images/drafts/ai-app-development-environment-variables-thumbnail.png)

## 想定読者

- AIアプリ開発を始めたばかりの人
- OpenAI APIキーをどこに置けばよいか分からない人
- `.env`、環境変数、本番環境の違いが曖昧な人
- GitHubに秘密情報を入れてしまわないか不安な人

## 読者の課題

AIアプリでは、画面、サーバー側API、AI APIが連携します。

このとき初心者が迷いやすいのが、APIキーの置き場所です。

たとえば、次のような疑問が出てきます。

- `.env` に書けば安全なのか
- GitHubにpushしてよいファイルはどれか
- VPSやDockerではどこに設定すればよいのか
- ブラウザ側のコードからAPIキーは見えないのか
- APIキーが漏れたら何が起きるのか

この記事では、AIアプリ開発に必要な範囲に絞って、環境変数の基本を整理します。

## 環境変数とは

環境変数は、アプリの実行環境に渡す設定値です。

コードの中に固定で書くのではなく、アプリを動かす場所ごとに外側から渡します。

AIアプリでよく使う環境変数には、次のようなものがあります。

| 環境変数の例 | 何に使うか | 秘密情報か |
| --- | --- | --- |
| `OPENAI_API_KEY` | OpenAI APIを呼ぶためのキー | はい |
| `DATABASE_URL` | データベース接続先 | 多くの場合はい |
| `NEXT_PUBLIC_SITE_URL` | サイトURLなど公開してよい設定 | いいえの場合が多い |
| `NODE_ENV` | 開発環境か本番環境かの判定 | いいえ |

重要なのは、すべての環境変数が秘密情報とは限らないことです。

ただし、APIキー、トークン、パスワード、秘密の接続URLは、基本的に外へ出してはいけません。

## なぜAPIキーをコードに直接書かないのか

APIキーをコードに直接書くと、次のようなリスクがあります。

| やってしまうこと | 起きる問題 |
| --- | --- |
| ソースコードにAPIキーを書く | GitHubにpushしたときに漏れる可能性がある |
| フロントエンドにAPIキーを書く | ブラウザから見える可能性がある |
| チャットやIssueにAPIキーを貼る | 共有先に秘密情報が残る |
| ログにAPIキーを出す | エラー調査用ログから漏れる可能性がある |

AI APIは、キーを持っている人がAPIを呼び出せる仕組みです。

キーが漏れると、意図しないAPI利用や料金発生につながる可能性があります。

そのため、AIアプリでは「APIキーはサーバー側だけで使う」「コードや画面に出さない」という考え方が基本になります。

## `.env` は何のために使うのか

`.env` は、ローカル開発で環境変数をまとめて書いておくためのファイルです。

たとえば、手元のPCで開発するときは次のように書きます。

```txt
OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxx
```

アプリ側では、コードにキーを直接書かず、環境変数として読み込みます。

```ts
const apiKey = process.env.OPENAI_API_KEY;
```

この形にしておくと、開発環境、本番環境、検証環境で別々の値を使いやすくなります。

ただし、`.env` は便利ですが、扱いを間違えると危険です。

| ファイル | 役割 | GitHubに入れてよいか |
| --- | --- | --- |
| `.env` | 実際の秘密情報を書く | 入れない |
| `.env.local` | ローカル専用の秘密情報を書く | 入れない |
| `.env.example` | 必要な環境変数名だけを共有する | 入れてよい |

`.env.example` には、実際のAPIキーではなく、次のように空欄やダミー値を書きます。

```txt
OPENAI_API_KEY=
```

これなら、プロジェクトに必要な設定項目だけを共有できます。

## AIアプリでの基本構成

AIアプリでは、APIキーをブラウザ側に置かず、サーバー側APIで読み込むのが基本です。

| 流れ | 役割 |
| --- | --- |
| 1. ユーザー | 画面に質問や文章を入力する |
| 2. ブラウザ | 入力内容をサーバー側APIへ送る |
| 3. サーバー側API | 環境変数から `OPENAI_API_KEY` を読み込む |
| 4. AI API | サーバーから送られた内容を処理する |
| 5. サーバー側API | AI APIの結果を受け取り、画面へ返す |
| 6. ブラウザ | 結果をユーザーに表示する |

この構成では、ブラウザはAI APIキーを直接持ちません。

ブラウザはサーバー側APIに入力内容を送ります。サーバー側APIが環境変数からAPIキーを読み込み、AI APIを呼び出します。

Next.jsでAIアプリを作る基本構成は、[Next.jsでAIアプリを作る基本構成](/articles/nextjs-ai-app-basic-architecture) でも整理しています。

## ローカル環境と本番環境の違い

環境変数は、アプリを動かす場所によって設定方法が変わります。

| 場所 | 例 | 設定方法の考え方 |
| --- | --- | --- |
| ローカルPC | 自分の開発環境 | `.env.local` などに設定する |
| VPS | 自分で管理するサーバー | サーバー上の `.env` やDocker Composeのenvで渡す |
| Vercelなど | ホスティング環境 | 管理画面のEnvironment Variablesに設定する |
| GitHub Actions | CI環境 | GitHub Secretsに設定する |

初心者が混乱しやすいのは、「ローカルで動いたのに本番で動かない」というケースです。

この場合、本番環境に同じ環境変数が設定されていないことがあります。

## VPSやDockerで考えること

VPSやDocker ComposeでAIアプリを動かす場合も、考え方は同じです。

大事なのは、コンテナの中でアプリが環境変数を読める状態にすることです。

たとえば運用時には、次のような観点で確認します。

| 確認項目 | 見るポイント |
| --- | --- |
| `.env` が存在するか | VPS上に必要な設定があるか |
| Docker Composeに渡っているか | コンテナ起動時に環境変数が渡されるか |
| アプリの再起動が必要か | 変更後に反映されているか |
| ログにキーが出ていないか | 秘密情報を出力していないか |

実際のコマンドは環境によって変わりますが、考え方は次の順番です。

```txt
1. VPS上で環境変数を設定する
2. アプリまたはコンテナを再起動する
3. アプリが環境変数を読めているか確認する
4. ログに秘密情報が出ていないか確認する
```

## よくある失敗

環境変数まわりでは、次の失敗がよく起きます。

| 失敗 | 原因 | 対策 |
| --- | --- | --- |
| ローカルでは動くが本番で失敗する | 本番環境にAPIキーが設定されていない | 本番側の環境変数を確認する |
| GitHubにAPIキーを入れてしまう | `.env` をignoreしていない | `.gitignore` と履歴を確認する |
| ブラウザからAPIキーが見える | フロントエンド側でキーを使っている | サーバー側APIに移す |
| APIキーがundefinedになる | 環境変数名が違う | 名前のスペルを確認する |
| 設定を変えたのに反映されない | アプリを再起動していない | 再起動して読み直す |

特に、`OPENAI_API_KEY` と `OPEN_AI_API_KEY` のような小さなスペル違いは見落としやすいです。

## 実装前チェックリスト

AI APIキーを扱う前に、次を確認します。

- [ ] APIキーをソースコードに直接書いていない
- [ ] APIキーをフロントエンドで使っていない
- [ ] `.env` や `.env.local` をGitHubに入れていない
- [ ] `.env.example` には実キーを書いていない
- [ ] 本番環境にも同じ環境変数名を設定している
- [ ] 設定変更後にアプリやコンテナを再起動している
- [ ] ログにAPIキーを出していない
- [ ] エラー時に秘密情報を画面へ表示していない

このチェックを通しておくと、AIアプリを公開するときの不安がかなり減ります。

## APIキーが漏れたかもしれない時

もしAPIキーをGitHub、チャット、ログ、スクリーンショットなどに出してしまった可能性がある場合は、まずキーを無効化または再発行することを考えます。

そのうえで、次を確認します。

1. どこにキーが出たか確認する
2. その場所からキーを削除する
3. 必要なら履歴やログの扱いを確認する
4. 新しいキーを環境変数に設定する
5. アプリを再起動する
6. 古いキーで動いていないことを確認する

「見えない場所に移したから大丈夫」と考えるのではなく、漏れた可能性があるキーは使い続けない方が安全です。

## まとめ

環境変数は、AIアプリ開発でAPIキーや設定値を安全に扱うための基本です。

最初は難しく見えますが、考え方はシンプルです。

- 秘密情報はコードに直接書かない
- ローカルでは `.env` や `.env.local` を使う
- 本番環境ではサーバーやホスティング側で設定する
- ブラウザ側にAPIキーを出さない
- 変更後はアプリを再起動して確認する

環境変数を理解しておくと、AI APIを使った小さなアプリから本番運用まで、より安全に進めやすくなります。

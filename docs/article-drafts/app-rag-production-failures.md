## 結論

RAGアプリが本番で失敗する原因は、回答生成だけではありません。

多くの場合、問題は「検索」「文脈」「回答」「評価」のどこかに分かれます。

まずは次のように切り分けます。

| 切り分け対象 | 確認すること |
| --- | --- |
| 文書更新 | 最新情報が取り込まれているか |
| chunk設計 | 検索しやすい単位で分割されているか |
| 検索品質 | 質問に対して正しい文書が上位に出るか |
| 文脈生成 | 余計な文書や不足した文書を渡していないか |
| 回答生成 | 参照元に基づいて回答しているか |
| 評価 | 改善前後を比べる質問セットがあるか |

## RAGはなぜデモではうまく見えやすいのか

RAGは、少数の文書と分かりやすい質問ではうまく動いて見えます。

しかし本番では、文書量、質問の揺れ、更新頻度、表現の違い、古い情報の混在が増えます。

| デモ環境 | 本番環境 |
| --- | --- |
| 文書が少ない | 文書が多く、古い情報も混ざる |
| 質問が想定通り | 曖昧な質問や言い換えが来る |
| 成功例を見せる | 失敗例も継続的に発生する |
| 手元で調整できる | 利用ログを見て改善する必要がある |

RAGでは、回答文だけを見ても原因は分かりません。

検索結果と参照元を見て、どこで失敗したかを確認します。

## 典型的な失敗

### 最新文書が取り込まれていない

本番では文書が更新されます。

FAQ、仕様書、社内ドキュメント、価格表、手順書などが更新されても、RAG側のインデックスが古いままだと、古い情報をもとに回答します。

確認する項目は次の通りです。

- 文書の最終更新日
- 取り込み処理の実行日時
- インデックスに入っている文書ID
- 削除済み文書が残っていないか

### chunkが検索に合っていない

chunkが長すぎると、検索結果に余計な情報が混ざります。

chunkが短すぎると、回答に必要な文脈が足りません。

| chunkの問題 | 起きること |
| --- | --- |
| 長すぎる | 関係ない文章までAIに渡る |
| 短すぎる | 前後関係が消えて回答できない |
| 見出しを失う | 何の説明か分からなくなる |
| 表や箇条書きを崩す | 重要な対応関係が失われる |

chunk設計は、検索品質と回答品質の両方に影響します。

### 検索結果が質問とずれている

RAGの失敗で多いのは、回答生成ではなく検索の失敗です。

質問に対して、上位に関係ない文書が出ている場合、AIは正しい回答を作りにくくなります。

確認すべきログは次の通りです。

| ログ項目 | 目的 |
| --- | --- |
| question | 実際の質問を見る |
| retrievedDocumentIds | 取得された文書を見る |
| scores | 検索スコアを見る |
| chunkTextPreview | 渡された文脈の概要を見る |
| answer | 最終回答を見る |
| sourceIds | 回答に使った参照元を見る |

検索結果を見ずにプロンプトだけ調整しても、根本原因を見落とします。

## 調査手順

本番でRAG回答が悪いときは、次の順番で調査します。

1. 実際の質問を1件選ぶ
2. その質問で検索された文書を確認する
3. 上位結果が正しいか確認する
4. AIに渡した文脈を確認する
5. 回答が文脈に基づいているか確認する
6. 同じ質問を改善前後で比べる

この順番で見ると、検索が悪いのか、渡す文脈が悪いのか、回答生成が悪いのかを分けやすくなります。

## 改善ポイント

RAG改善では、いきなりモデルやプロンプトを変える前に、検索側を確認します。

| 問題 | 改善例 |
| --- | --- |
| 最新文書がない | 定期取り込み、差分取り込みを入れる |
| 検索結果がずれる | メタデータ、フィルタ、クエリ整形を見直す |
| chunkが悪い | 見出し単位、段落単位、表の扱いを調整する |
| 参照元が不明 | 回答にsourceを表示する |
| 改善効果が不明 | 評価質問セットを作る |

## 評価質問セットを持つ

RAGを本番で改善するには、評価用の質問セットが必要です。

たとえば、よくある質問、失敗しやすい質問、最新情報を問う質問を10〜30件用意します。

| 質問タイプ | 例 |
| --- | --- |
| 基本質問 | 仕様の概要を聞く |
| 詳細質問 | 条件や例外を聞く |
| 最新質問 | 更新後の内容を聞く |
| 曖昧質問 | ユーザーが雑に聞く |
| 失敗再現質問 | 過去に間違えた質問を再利用する |

改善前後で同じ質問を使うと、RAGの品質が上がったのか判断しやすくなります。

## 本番投入前チェックリスト

RAGアプリを公開する前に、次を確認します。

- [ ] 文書の更新日と取り込み日時を確認できる
- [ ] 検索結果の上位文書をログで見られる
- [ ] 回答に参照元を表示できる
- [ ] chunk設計の方針がある
- [ ] 評価用の質問セットがある
- [ ] 古い文書や削除済み文書が残らない
- [ ] 検索失敗と回答失敗を分けて調査できる
- [ ] ユーザーに誤回答の可能性を適切に伝えられる

## まとめ

RAGアプリの本番失敗は、回答生成だけを見ても解決しません。

文書更新、chunk設計、検索結果、参照元、評価質問セットを分けて確認する必要があります。

まずは、実際の質問と検索結果をログに残し、検索の失敗なのか回答生成の失敗なのかを切り分けてください。

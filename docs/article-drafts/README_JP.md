# 記事下書き

このディレクトリには、まだ公開しない記事下書きを置きます。

このディレクトリ内のMarkdownファイルは、Next.jsサイトには読み込まれません。
公開する記事だけを、確認後に `apps/web/src/content/articles` へ移動します。

## レビュー運用

おすすめの確認手順は以下です。

1. 新しい記事案は `docs/article-drafts` に置く
2. タイトル、想定読者、構成、事実関係、公開優先度を確認する
3. 承認されるまで、このディレクトリ内で下書きを修正する
4. 公開する記事だけを `apps/web/src/content/articles` へ移動する
5. `apps/web` で `npm run build` を実行する
6. commit、push後、`scripts/deploy-vps.ps1` でVPSへ反映する

## 公開品質基準

公開候補の記事は、最低限以下を満たす状態にします。

- 1500〜2500字程度の本文量がある
- 冒頭に「この記事で分かること」がある
- 想定読者が明確である
- 結論が早めに書かれている
- 表、チェックリスト、Mermaid図のいずれかを複数含む
- 読者がそのまま使える手順、判断基準、確認項目がある
- 関連記事または関連ツールへの導線がある
- ニュース記事では、事実、解釈、開発者への示唆が分かれている
- 事実確認が必要な主張には `needs_fact_check: true` を付ける

## 推奨記事構成

基本構成は以下を推奨します。

1. この記事で分かること
2. 想定読者
3. 結論
4. 全体像または図解
5. 用語整理または比較表
6. 具体的な手順
7. よくある失敗と対策
8. チェックリスト
9. 関連記事・関連ツール導線
10. まとめ

## レビューチェックリスト

- 3本柱のいずれかに合っている
- タイトルが具体的で、検索ユーザーに役立つ
- 想定読者が明確である
- 手順、例、判断基準など、実用的な内容が含まれている
- 表や図解によって、視覚的に理解しやすくなっている
- ニュース記事では、事実、解釈、開発者への示唆が分かれている
- 明示的に計画していない限り、SaaS、決済、ログイン、ダッシュボード機能を前提にしない

## 結論

AI生成アプリが本番で落ちる主な理由は、生成コードそのものが悪いからだけではありません。

開発環境では見えなかった条件が、本番環境で一気に表面化するためです。

特に多いのは、次の5つです。

| 原因 | 本番で起きること |
| --- | --- |
| 環境変数の未設定 | APIキーや接続先が読めずに失敗する |
| 型やnullの扱い不足 | 本番データで例外が出る |
| API失敗時の処理不足 | 外部APIの失敗がそのまま画面崩れになる |
| 非同期処理の競合 | 保存前に表示する、二重送信する |
| ログ不足 | どこで落ちたか分からない |

## なぜ開発環境では動いて見えるのか

開発環境では、条件が単純です。

自分の入力だけで試す、データ量が少ない、APIキーが手元にある、エラーが出てもすぐコンソールを見られる、といった前提があります。

本番環境では、この前提が変わります。

| 開発環境 | 本番環境 |
| --- | --- |
| 入力が限定的 | 予想外の入力が来る |
| 自分だけが使う | 複数ユーザーが使う |
| `.env` が手元にある | デプロイ先に設定が必要 |
| コンソールをすぐ見られる | ログ設計がないと追えない |
| 失敗しても影響が小さい | ユーザー体験に直結する |

AI生成コードは、成功パターンの実装は速い一方で、失敗パターンの扱いが薄くなることがあります。

## よくある原因

### 環境変数が本番にない

開発環境では `.env.local` にAPIキーがあり、問題なく動くことがあります。

しかし、本番環境に同じ環境変数が設定されていないと、API呼び出しは失敗します。

```ts
const apiKey = process.env.OPENAI_API_KEY;

if (!apiKey) {
  throw new Error("OPENAI_API_KEY is not set");
}
```

本番では、環境変数が空の場合に分かりやすいエラーを出す設計が必要です。

### nullやundefinedを想定していない

AI生成コードは、サンプルデータでは動く形で作られることがあります。

本番データでは、値が空、配列が空、想定外の形式、APIレスポンスが欠けている、といったことが起きます。

```ts
const title = response.article.title;
```

このようなコードは、`response.article` が存在しない場合に落ちます。

最低限、外部APIやDBから来る値は信用しすぎない設計にします。

### API失敗時のcatchがない

AI API、DB、外部サービスは常に成功するとは限りません。

ネットワーク、レート制限、タイムアウト、認証失敗などがあります。

```ts
try {
  const result = await callAiApi(input);
  return Response.json({ result });
} catch (error) {
  console.error("AI API failed", error);
  return Response.json(
    { error: "AIの処理に失敗しました。時間をおいて再試行してください。" },
    { status: 500 },
  );
}
```

ユーザーには分かりやすいエラーを返し、開発者向けにはログを残します。

## 確認コマンド例

AI生成アプリを公開する前に、最低限次の確認を行います。

```bash
npm run build
```

型チェックやビルド時のエラーを確認します。

```bash
npm run lint
```

lintスクリプトがある場合は、未使用変数や危険な書き方を確認します。

```bash
git diff --check
```

不要な空白や差分の基本的な崩れを確認します。

本番に近い環境で起動できる場合は、環境変数を設定した状態で主要導線を確認します。

## ログで見るべき項目

本番障害では、ログの粒度が重要です。

| ログ項目 | 目的 |
| --- | --- |
| requestId | 1回の処理を追跡する |
| route | どのAPIやページで失敗したか見る |
| status | 成功、失敗、タイムアウトを分ける |
| durationMs | 遅い処理を見つける |
| errorName | エラー種別を見る |
| inputLength | 長すぎる入力が原因か見る |

ログにAPIキー、個人情報、入力全文をそのまま出してはいけません。

## 本番投入前チェックリスト

公開前に、次を確認します。

- [ ] 本番環境に必要な環境変数を設定した
- [ ] APIキーをブラウザ側に置いていない
- [ ] `npm run build` が通る
- [ ] 外部API失敗時のcatchがある
- [ ] nullやundefinedを想定している
- [ ] 空入力、長すぎる入力をチェックしている
- [ ] エラー時にユーザーへ分かりやすい表示を返す
- [ ] 本番ログで失敗箇所を追える
- [ ] APIキーや個人情報をログに出していない

## AI生成コードをレビューするときの観点

AI生成コードは、生成された瞬間ではなく、レビューと検証を通して品質を上げます。

| 観点 | 確認すること |
| --- | --- |
| 入力 | 空、長すぎる、想定外の形式に対応しているか |
| 外部API | 失敗、タイムアウト、レート制限に対応しているか |
| 状態管理 | 二重送信や競合が起きないか |
| 環境差 | ローカルと本番で必要な設定が一致しているか |
| ログ | 原因調査に必要な情報が残るか |

「動いた」だけで終わらせず、「失敗しても壊れ方が分かる」状態にします。

## まとめ

AI生成アプリが本番で落ちる原因は、環境変数、型、API失敗、非同期処理、ログ不足に集中しがちです。

AIコーディングツールを使うほど、ビルド、入力チェック、エラー処理、ログ確認を標準化する必要があります。

本番公開前には、成功パターンだけでなく、失敗パターンを確認してください。

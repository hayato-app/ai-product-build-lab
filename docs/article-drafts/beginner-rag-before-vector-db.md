## この記事で分かること

- RAGの基本的な仕組み
- ベクトルDBを使う前に考えるべきこと
- chunk、embedding、検索、回答生成の役割
- RAGで失敗しやすいポイント

## 想定読者

- RAGという言葉は知っているが仕組みが分からない人
- ベクトルDBを入れれば解決すると思っている人
- 社内文書検索やFAQチャットを作りたい人

## 結論

RAGは、ベクトルDBを入れるだけではうまくいきません。重要なのは、文書をどう分けるか、どの情報を検索するか、検索結果をどう回答に渡すかです。

RAGの品質は、モデル性能よりも「検索で正しい情報を取れているか」に大きく左右されます。

## RAGの基本構造

```mermaid
flowchart LR
  Docs["文書"] --> Chunk["chunkに分割"]
  Chunk --> Embed["embedding化"]
  Embed --> VectorDB["検索基盤に保存"]
  Question["質問"] --> Search["関連chunkを検索"]
  VectorDB --> Search
  Search --> Prompt["検索結果をプロンプトに追加"]
  Prompt --> Answer["AIが回答"]
```

## 用語整理

| 用語 | 意味 | 注意点 |
| --- | --- | --- |
| chunk | 文書を分割した単位 | 大きすぎても小さすぎてもよくない |
| embedding | 文章を数値ベクトルにしたもの | モデルによって検索結果が変わる |
| ベクトルDB | embeddingを保存・検索する基盤 | 入れれば解決するものではない |
| 検索結果 | 質問に近い文書断片 | 正しい文書が取れているか確認が必要 |
| 生成プロンプト | 検索結果を含めたAIへの指示 | 長すぎるとコストと速度に影響 |

## chunk設計の考え方

chunkはRAGの品質に直結します。1つのchunkに複数の話題が混ざると、検索結果に不要な情報が入りやすくなります。逆に短すぎると、回答に必要な文脈が失われます。

最初は、見出し単位や意味のまとまり単位で分けるのが扱いやすいです。FAQなら1問1答、マニュアルなら見出しごと、議事録なら議題ごとに分けると確認しやすくなります。

## RAGでよくある失敗

| 失敗 | 原因 | 対策 |
| --- | --- | --- |
| 関係ない回答をする | 検索結果がずれている | 検索結果ログを見る |
| 古い情報で回答する | 文書更新が反映されていない | 再取り込みの仕組みを作る |
| 回答が長くて曖昧 | 渡す文脈が多すぎる | 上位件数を絞る |
| 根拠が分からない | 参照元を表示していない | 出典リンクや文書名を出す |
| コストが高い | 検索結果を渡しすぎる | chunkと件数を調整する |

## 最初に作るRAGのおすすめ構成

```text
文書: Markdownまたはテキスト
分割: 見出し単位
検索件数: 上位3〜5件
回答: 検索結果に基づいて回答
表示: 回答 + 参照元
```

最初から複雑なランキングや再検索を入れる必要はありません。まずは検索結果が妥当かを目で確認できる状態にすることが大切です。

## 評価用の質問セットを作る

RAGを改善するには、毎回同じ質問で確認できるようにします。

| 質問 | 期待する参照文書 | 期待する回答 |
| --- | --- | --- |
| 料金プランは？ | pricing.md | プラン一覧を説明 |
| 解約方法は？ | account.md | 解約手順を説明 |
| API制限は？ | api-limits.md | 上限と注意点を説明 |

このような表を作ると、chunkや検索設定を変えたときに改善したか判断しやすくなります。

## まとめ

RAGは、ベクトルDBを導入するだけでは完成しません。文書設計、chunk設計、検索品質、参照元表示、評価質問をセットで考えることで、実用に近いRAGアプリになります。
